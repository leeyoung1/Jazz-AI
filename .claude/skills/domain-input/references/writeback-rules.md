# Writeback Rules Reference

### Phase 3：规整与结构化

**3.1 信息提取**

从用户输入（无论来源是文档、文件还是交互式输入）中提取以下维度的信息：

1. **业务域全景**：系统定位、核心场景、用户角色
2. **核心业务规则**：校验、状态流转、计算、权限、约束
3. **领域术语表**：术语 + 定义 + 同义词/近义词
4. **产品架构**：各端关系、外部依赖、数据流向
5. **关键业务流程**：端到端的核心流程描述
6. **补充说明**：历史背景、特殊约定、业务方向

**图片信息提取原则**（适用于文档/文件中包含图片的场景）：

- 流程图 / 时序图 → 提取到「关键业务流程」章节，描述步骤、角色、关键节点
- 架构图 / 部署图 → 提取到「产品架构」章节，描述各端和外部依赖
- 状态机图 → 提取到「核心业务规则 - 状态流转规则」章节
- 数据模型图 / ER 图 → 提取核心实体和关系，补充到「领域术语表」
- 截图 / UI 图 → 从界面元素推断业务场景和用户角色，补充到「业务域全景」
- 所有从图片提取的内容均标注 `[来源：图片]`

**3.2 质量检查**

对提取的信息进行检查：

- 术语定义是否清晰（模糊的标注 `[待澄清]`）
- 规则描述是否有矛盾（有矛盾的标注 `[矛盾，请确认]`）
- 流程描述是否完整（缺少关键节点的标注 `[不完整]`）
- 与 AGENTS.md 中的术语是否一致（不一致的标注映射关系）

**3.3 增量合并（如已有 domain.md）**

如果是增量更新模式：

- 新增内容：追加到对应章节
- 重复内容：保留已有内容，新内容标注 `[新增]`
- 冲突内容：保留两个版本，标注 `[冲突，请确认]`，在确认环节让用户决策
- 补充/丰富已有内容：与已有内容合并，标注 `[更新]`

### Phase 4：人工确认

**4.1 展示规整后的内容**

完整展示 domain.md 的内容：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 DOMAIN KNOWLEDGE — 业务领域知识（规整后）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

{完整的 domain.md 内容}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**4.2 确认写入**

```
AskUserQuestion:
  question: |
    请审核以上业务领域知识文档。确认后将保存到 .vibe/domain.md。

    如果需要调整，请直接说明修改内容。
  options:
    - 确认写入
    - 需要调整（请在回复中说明）
    - 暂不保存
```

**4.3 处理冲突项（如有）**

如果存在 `[冲突，请确认]` 或 `[待澄清]` 的标注项，逐项确认。
