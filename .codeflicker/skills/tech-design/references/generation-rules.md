# Generation Rules Reference

### Phase 1A：加载上下文

1. 读取 `.prd/{PROJECT_NAME}/03-context/context.md` — 上下文分析报告
2. 读取 `.prd/{PROJECT_NAME}/01-prd/prd.md` — PRD 摘要
3. 读取 `.vibe/AGENTS.md`（或根目录 `AGENTS.md`）— 仓库规范（如存在）
4. 读取 `.prd/{PROJECT_NAME}/shared/scope_map.json` — 范围定位地图
5. 读取 `.vibe/index.md` — 代码模板索引（如存在）
6. 读取 `.vibe/interfaces/index.md` — 后端接口图谱总索引（如存在）
   - 只有当功能点命中特定功能域时，才读取对应 `.vibe/interfaces/domains/*.md`
7. 读取 `.prd/{PROJECT_NAME}/shared/question_backlog.json` — 残留问题待办（如存在）

如果 context.md 不存在：
- 告知用户：「未找到上下文分析报告。请先执行 /explore --project {PROJECT_NAME}」
- 停止工作流

### Phase 1.5A：残留问题决策 + 实现偏好收集

**1.5A.1 全量澄清 question_backlog 残留项（强阻断）**

如果 `question_backlog.json` 存在，筛选 `status` 为 `open` 或 `partial` 且 `ask_stage` 为 `tech-design` 的问题：

1. 先整理所有残留问题的完整清单
2. 逐条使用 AskUserQuestion 向用户提问，每次只提一个问题：

```
AskUserQuestion:
  question: |
    【方案前确认 {序号}/{总数}】
    {问题描述}
    
    来源：{来源阶段/功能点}
    不确认影响：{影响说明}
    {如果有代码证据：代码证据: {文件路径}:{方法名}}
  options:
    - {选项A}
    - {选项B}
    - 其他（请说明）
```

3. 所有问题用户回答完毕后，立即将 question_backlog.json 中对应项 status 更新为 `resolved`，记录 resolution
4. **【强阻断】所有残留问题必须全部获得用户回答后，才能进入 Phase 1.5A.1b 接口二次确认、Phase 1.5A.2 偏好收集和 Phase 2A 方案生成**
   - 不允许跳过任何残留问题
   - 不允许自行假设并继续（证据只有代码和用户信息）

无论是否存在残留问题，完成 1.5A.1 后都必须进入 Phase 1.5A.1b 接口二次确认。

**1.5A.1b 接口二次确认（强阻断）**

检查 `scope_map.json` 中所有 `confirmed_interfaces`。如果存在 `status = pending` 的项，先逐条向用户确认具体接口或新增位置，更新为 `confirmed` 或 `confirmed_location` 后再继续。

如果存在 `status = confirmed` 或 `status = confirmed_location` 的项：

1. 向用户展示完整接口确认清单（区分 action 类型：modify 为修改已有、new 为新增、reference 为参考）：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 接口确认清单（来自 explore 阶段）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

| 功能点 | 接口/位置 | 动作 | 确认时间 |
|--------|-----------|------|----------|
| ...    | pkg.Service/MethodA 或 service/proto | 修改/新增/参考 | ... |

注：new（新增）接口已确认归属 service/proto，接口结构（入参/出参/错误码）将在 Phase 2A 生成方案时设计。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

2. 使用 AskUserQuestion 确认：

```
AskUserQuestion:
  question: |
    【接口二次确认】
    以上是 explore 阶段确认的接口清单（修改/参考接口含 RPC 方法名，新增接口含 service/proto 归属位置）。
    请确认是否有遗漏的接口？确认后将以此清单生成方案。
    （新增接口在后续生成方案时将设计具体入参/出参/错误码）
  options:
    - 确认无误，继续生成方案
    - 有遗漏（请说明遗漏的接口）
```

3. **强制规则**：
   - 即使清单看起来完整，也必须向用户确认，不允许跳过
   - 用户确认无遗漏 → 继续 Phase 1.5A.2 收集实现偏好
   - 用户指出遗漏 → 补充到清单，再次确认后继续
4. 如果 `scope_map.json` 中不存在 `confirmed_interfaces`（无接口改造需求），跳过本步骤
5. 如果 `scope_map.json` 中只有 `interface_candidates` 而没有 `confirmed_interfaces`，必须基于候选项向用户确认后写入 `confirmed_interfaces`，不得直接跳过

如果无接口清单需要确认：直接进入 Phase 1.5A.2 偏好收集

**1.5A.2 收集实现偏好（复杂度 ≥ 中等时必须执行）**

根据 context.md 的改动点类型，自动识别灰色地带并收集用户偏好：
- **UI/前端类改动** → 问布局偏好、交互方式、空状态处理
- **API/接口类改动** → 问响应格式、错误处理风格、版本策略
- **数据类改动** → 问迁移策略、兼容方案、数据清洗方式
- **跨模块改动** → 问拆分策略、灰度方案

使用 AskUserQuestion 展示 2-5 个偏好选项（根据改动复杂度自适应）：

```
AskUserQuestion:
  question: |
    在生成技术方案前，有以下实现偏好需要确认：

    1. {偏好问题1}
       A) {选项A}  B) {选项B}
    2. {偏好问题2}
       A) {选项A}  B) {选项B}

    请逐条选择，或回复「默认」使用推荐方案。
  options:
    - 逐条选择（请在回复中说明）
    - 使用默认推荐
```

将用户偏好记录，在 Phase 2A 生成方案时作为约束输入。

**跳过条件**：
- **L1（Lite Flow）模式直接跳过偏好收集**（已在 prd-input 阶段确认走轻量流程，不再追加偏好问题）。
- 非 Lite 流程：只有当改动点 ≤ 3 **且** 复杂度判定为"简单"时，才允许跳过偏好收集。中等及以上复杂度**必须**收集偏好，不得跳过。

**如果 AskUserQuestion 不可用**：用文本方式列出偏好选项，等待用户回复后继续。不得在用户未回复的情况下使用默认值。

### Phase 2A：生成技术方案

基于 context.md 中的改动点分析，生成技术方案。

**方案详细程度自适应规则：**

| 需求复杂度 | 判断条件 | 方案详细程度 |
|-----------|---------|------------|
| 简单 | 改动点 ≤ 3，无跨模块改动，无高风险项 | 精简：接口契约 + 改造点 + 依赖变更 |
| 中等 | 改动点 4-10，或涉及 2-3 个模块 | 标准：接口契约 + 改造点 + 数据模型 + 流程说明 |
| 复杂 | 改动点 > 10，或跨 3+ 模块，或有高风险项 | 完整：接口契约 + 改造点 + 时序说明 + 数据模型 + 影响分析 + 风险评估 + 兼容方案 |

**方案核心内容（按复杂度递增）：**

#### 必选内容（所有复杂度都包含）

1. **方案概要**
   - 需求一句话总结
   - 改造策略概述（新增/修改/重构）
   - 预计影响范围

2. **接口契约设计**
   - **优先级规则**：
     1. 优先读取 `scope_map.json` 的 `confirmed_interfaces`：如果存在 `status = confirmed` 或 `status = confirmed_location` 的接口/位置，将其直接作为接口契约的输入源，不再重新读取 domain file 推断接口身份
     2. 降级流程（仅当 `confirmed_interfaces` 为空、不存在或全为 `pending`）：先按功能点锚定接口功能域，参考 `.vibe/interfaces/index.md` 或 context.md 中的候选领域证据；命中特定功能域后，再读取对应 domain file 作为接口判断依据
     3. **硬规则**：降级流程中如果接口图谱或代码上下文无法唯一确定要修改的具体接口（含 RPC 方法全限定名），必须先向用户确认，禁止猜测后直接生成方案
   - 参考接口列表（只作为语义或实现参照，不直接改）
   - 修改接口列表（原接口 ID、变更内容、兼容性说明，必须与 `confirmed_interfaces` 中 `action = modify` 的接口一致）
   - 新增接口列表（路径、方法、入参、出参、错误码，以及为什么无法复用既有接口，必须落在 `confirmed_interfaces` 中 `action = new` 的 service/proto 位置下）
   - 删除接口列表（原接口、替代方案、灰度策略，必须与 `confirmed_interfaces` 中 `action = delete` 的接口一致）

3. **代码改造点**
   - 逐功能点列出具体的文件级改动说明
   - 优先说明推荐复用的模板/样例，以及基于这些参考物需要做的差量实现
   - 每个改动点关联到 context.md 中的分析结果
   - 标注遵循的链路模板

#### 标准内容（中等及以上复杂度包含）

4. **数据模型变更**
   - 新增表/字段
   - 修改字段含义或类型
   - 数据迁移方案（如需要）

5. **流程说明**
   - 核心流程的步骤描述
   - 关键分支逻辑
   - 与现有流程的差异对比

#### 完整内容（高复杂度包含）

6. **时序与交互说明**
   - 关键链路的交互步骤
   - 同步/异步调用关系
   - 超时和重试策略

7. **影响分析**
   - 对上下游系统的影响
   - 性能影响评估
   - 兼容性方案（新老并存策略）

8. **风险评估与应对**
   - 技术风险点
   - 缓解措施
   - 回滚方案
