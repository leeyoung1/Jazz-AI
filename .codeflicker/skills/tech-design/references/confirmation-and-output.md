**3.2 确认方案（必须等待用户明确回复）**

优先使用 AskUserQuestion：

```
AskUserQuestion:
  question: |
    请审核以上技术方案（含模板使用清单和接口契约判断清单）。接口图谱用于确认每个功能点命中的功能域，以及是否需要参考、修改或新增接口。确认后将保存技术方案文档。
    如果需要调整，请直接说明修改内容（包括模板选择意见）。
  options:
    - 确认方案
    - 需要调整（请在回复中说明修改内容）
    - 模板选择有问题（请说明哪个改造点应使用不同的模板）
```

**如果 AskUserQuestion 不可用（降级策略）：**

不得自行将方案标记为已确认。必须用文本方式向用户展示：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 TECH DESIGN — 等待确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

请审核以上技术方案，并明确回复：
- 回复「确认方案」→ 保存方案文档，进入下一步
- 回复「需要调整」+ 修改内容 → 修改后重新展示
- 不回复或回复含糊 → 不保存，继续等待
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**用户未明确回复"确认"或"同意"之前，不得保存 tech_design.md 的 confirmed 状态。**

**3.3 处理确认结果**

- **用户确认** → 保存 tech_design.md（将 `pending_human_confirm` 设为 `false`，记录 `stakeholder_confirmed_at`）；**此时更新 question_backlog.json**：将 Phase 1.5A 中已澄清的残留问题确认保留为 `resolved` 状态
- **用户要求调整** → 听取用户的具体修改意见，判断：
  - 如果调整涉及新的不确定项（无法从代码或已有信息推断）→ 先通过 AskUserQuestion 向用户逐条澄清新的不确定项，再回到 Phase 2A 重新生成
  - 如果调整内容明确（用户已给出具体修改方向）→ 直接回到 Phase 2A 重新生成
  - 重新生成后重新执行 Phase 3 展示和确认（可循环多次，无次数限制）
  - 重新生成期间不更新 question_backlog（避免循环中状态不一致）
- **用户反馈模板有问题** → 记录用户指定的模板意见，回到 Phase 2A 重新生成方案（使用用户指定的模板），重新执行 Phase 3 确认
- **用户要求大幅重写** → 回到 Phase 2A 重新生成，带上用户的调整意见；不更新 question_backlog
