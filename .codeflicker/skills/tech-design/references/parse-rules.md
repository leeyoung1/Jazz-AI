# Parse Rules Reference

## 解析分支

### Phase 1B：获取已有方案

**1.1 从URL获取**

如果提供了 `--url`：
- 使用可用的文档抓取工具获取方案内容
- 优先级同 prd-input：docs_read → MCP → curl
- 抓取失败时告知用户，不继续

**1.2 从文件获取**

如果提供了 `--file`：
- 直接读取本地文件

**1.3 校验内容**

检查获取到的内容是否是技术方案文档。最低要求包含以下任一：
- 接口设计/契约内容
- 改造点/改动说明
- 数据模型变更
- 流程描述

如果内容明显不是技术方案（如纯需求文档、纯讨论记录）：
- 告知用户：「获取到的内容不像是技术方案文档。请确认链接/文件是否正确。」
- 停止工作流

### Phase 2B：解析与对齐

1. **解析方案结构** — 提取接口契约、改造点、数据模型、流程说明等结构化信息
2. **对齐代码库** — 将方案中提到的改动点与实际代码进行比对：
   - 方案提到的文件是否存在
   - 方案提到的方法签名是否与实际代码一致
   - 方案的假设是否与代码实际匹配
3. **补充缺失项** — 如果方案中缺少必要信息（如未定义错误码、未说明兼容策略），补充并标注 `[补充]`
4. **冲突标记** — 如果方案与代码现状有冲突，标注 `[冲突]` 并说明

解析完成后，同样输出为 `.prd/{PROJECT_NAME}/04-design/tech_design.md` 格式（统一结构）。
