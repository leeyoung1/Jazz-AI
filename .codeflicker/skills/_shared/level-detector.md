# Level Detector — 需求复杂度档位判定

> 被 `/prd-input` 引用，在需求录入阶段判定本次需求走 **Lite Flow** 还是 **Full Flow**。
> 判定结果写入 `.prd/{PROJECT_NAME}/shared/level.json`，供 hooks（policy-engine）与下游 skill（tech-design、tdd）消费。

## 档位定义

| 档位 | 流程 | 适用场景 | 用户确认 |
|------|------|---------|---------|
| **L1（Lite Flow）** | `prd-input → tech-design → tdd` | 改动 ≤5 文件、< 200 行、不跨服务、无新增接口（仅改已有接口实现或纯内部逻辑） | 1 道（tech-design 方案确认） |
| **L2（Standard）** | 完整 7 阶段 | 6-15 文件、200-800 行、单服务跨模块、新增 0-2 个接口 | 2 道（tech-design + tasks） |
| **L3（Full）** | 完整 7 阶段 + `--full` TDD | 跨服务、数据迁移、高风险、新增多个接口或接口契约变更 | 2 道（tech-design + tasks） |

> L2 与 L3 都走完整流程，区别只在 TDD 严格度与方案详细度。**只有判定为 L1 时才写 `level: "L1"` 并启用 Lite Flow**；L2/L3 写对应值但等同现状完整流程（hooks 只对 L1 特殊放行）。

## 判定信号（优先级从高到低）

1. **接口变更（最高权重）**：需求需要**新增** RPC/HTTP 接口，或变更接口契约（入参/出参/错误码）→ 至少 L2，不得判 L1。
   - 仅修改已有接口的内部实现、不改契约 → 不触发升档。
2. **跨服务**：改动涉及 2 个及以上服务/仓库 → 至少 L2。
3. **高风险**：数据迁移、灰度、资金/权限敏感、状态机变更 → 至少 L2，复杂时 L3。
4. **改动文件数**：≤5 → L1 候选；6-15 → L2；>15 → L3。
5. **改动行数（辅助）**：<200 → L1 候选；200-800 → L2；>800 → L3。

## 判定规则

1. 先看强信号（接口变更、跨服务、高风险）：命中任一 → 至少 L2。
2. 强信号都不命中时，按文件数 + 行数取**较高档**（如 4 文件但 600 行 → 取行数对应的 L2）。
3. **边界值保守上调**：5/6 文件之间、180-220 行之间等边界，默认上调一档。
4. 信息不足以判断时，默认 L2（完整流程更安全）。

## 判定来源

- PRD 原文规模与关键词：「迁移」「灰度」「新增接口」「跨服务」「对接」等 → 升档信号。
- 若已有 `scope_map.json`：`affected_files` 数量、`confirmed_interfaces` 是否含 `action = new`。
- 用户显式入口（`/lite-prd` 或 `--level Lx`）：**优先级最高**，覆盖自动判定（`/lite-prd` 直接确定 L1；`--level` 仍经 AskUserQuestion 二次确认）。
