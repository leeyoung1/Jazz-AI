# Domain Input — 业务领域知识注入

用于把业务领域知识、产品背景、术语、规则、流程和架构关系注入 `.vibe/domain.md`，作为后续需求理解和代码探索的业务基线。

## Working Rules

- 先确认当前模型是否具备图片识别能力；遇到图片、流程图、原型图时必须按 `references/input-handling.md` 处理。
- 先加载已有 `.vibe/domain.md`，再获取新增输入，不得直接覆盖已有知识。
- 支持文本、文件路径、文档链接、图片和混合输入；不同来源的处理细则见 `references/input-handling.md`。
- 只沉淀稳定业务知识，不把一次性需求实现细节写成领域规则。
- 写入前必须展示结构化结果并获得用户确认。
- 如果用户只想补充片段知识，应合并到对应章节，不重建整份文档。

## 参数

- `<输入>`：业务知识文本、文件路径、文档链接或图片说明。
- `--name <domain>`：可选，用于标记业务域名称。
- `--append`：追加/合并到已有 domain。
- `--replace`：仅在用户明确确认后替换已有内容。

## 工作流

### Phase 0：模型能力确认

执行前读取 `references/input-handling.md`，确认是否能直接识别图片；不能识别时向用户说明并要求提供文字版信息。

### Phase 1：加载已有知识

检查 `.vibe/domain.md` 是否存在；若存在，将其作为高优先级基线，后续只做补充、修订或结构化合并。

### Phase 2：获取输入

按输入类型读取用户提供的文本、文件、文档链接或图片内容；详细分支和失败处理见 `references/input-handling.md`。

### Phase 3：规整与结构化

按 `references/domain-template.md` 的结构整理业务域全景、核心规则、术语、产品架构、关键流程和补充说明。

### Phase 4：人工确认

按 `references/writeback-rules.md` 展示新增、修改、冲突和不确定项；用户确认前不得写入 `.vibe/domain.md`。

### Phase 5：写入与输出

确认后按 `references/writeback-rules.md` 合并或替换 `.vibe/domain.md`，并输出更新摘要和下一步建议。
