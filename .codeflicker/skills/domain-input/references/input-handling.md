# Input Handling Reference

## Phase 0: 模型能力确认（最高优先级）

> **⚠ 模型要求**：本阶段可能需要识别文档或文件中的图片（架构图、流程图、状态机等）。请确保当前使用的是**多模态模型**（如 Claude Sonnet 4.6 或同等级别），否则图片中的业务信息将无法被正确提取。

**在执行任何其他步骤之前，必须先询问用户确认模型能力。**

使用 AskUserQuestion 询问：

```
AskUserQuestion:
  questions:
    - question: |
        本阶段可能需要识别文档或文件中的图片（架构图、流程图、状态机等）。
        请确认当前使用的是多模态模型吗？
      header: 模型确认
      options:
        - label: 是多模态模型（推荐）
          description: 当前模型支持图片识别，可继续执行
        - label: 不是多模态模型
          description: 图片无法识别，将跳过图片内容提取
      multiSelect: false
```

根据用户回答处理：
- **是多模态模型**：继续执行后续 Phase
- **不是多模态模型**：跳过图片识别步骤，并在结果中标注「未进行图片识别」

接收人工提供的业务领域知识，规整为结构化文档，建立 AI 对业务域的基础认知。

## Working Rules

- 本阶段以**人工输入为主、AI 规整为辅**
- 不进入代码分析，不做技术方案设计
- 输出格式必须稳定，确保下游阶段可靠解析
- 支持增量更新：多次运行可叠加知识，不覆盖已有内容
- AI 规整时发现信息模糊或矛盾，标注 `[待澄清]` 而非自行推断
- 鼓励用户提供具体例子和场景，而非抽象描述

## 参数

调用时传入参数：

- `$ARGUMENTS` 格式：`[--url <文档URL>] [--file <本地文件路径>] [--append]`
  - `--url`：在线文档链接（可选，如内部 Wiki、飞书文档等）
  - `--file`：本地文件路径（可选，支持 Markdown / 文本文件）
  - `--append`：增量追加模式（可选，默认行为：如已有 domain.md 则增量合并）
  - 无参数时：进入交互式输入模式，逐维度引导用户补充
- 示例：
  - `/domain-input`（交互式引导输入）
  - `/domain-input --url https://wiki.xxx.com/biz-overview`（从文档提取）
  - `/domain-input --file ./docs/business-rules.md`（从本地文件提取）

## 工作流

### Phase 1：加载已有知识

**1.1 检查已有 domain.md**

读取 `.vibe/domain.md`（如存在）。

- 如果存在：进入增量更新模式，后续新输入合并到已有内容中
- 如果不存在：进入全新创建模式

**1.2 加载辅助参考**

按需读取（如有则读取，缺失不阻塞）：

1. `.vibe/AGENTS.md` — 仓库工程规范（如有，用于交叉对照术语）
2. `.archive/index.md` — 历史归档索引（如有，用于识别已沉淀的业务知识）

### Phase 2：获取输入

根据调用方式执行对应分支：

**2.1 从 URL 获取（--url）**

使用可用的文档抓取方式（浏览器、`curl`/`wget` 或粘贴正文）将文档下载到本地，**必须**按以下规则指定输出路径：

- 从 URL 中提取文档标题（或 docId）作为文件名，输出到 `docs/domain/<文档标题>/<文档标题>.md`
- 图片一并下载到 `docs/domain/<文档标题>/images/` 目录下
- **禁止**让工具自动决定输出路径，必须显式写入 `docs/domain/<文档标题>/<文档标题>.md`

然后按以下顺序执行，**每一步都必须完成，不得跳过**：

**步骤一：读取 md 文件**

用 `read` 工具读取 `docs/domain/<文档标题>/<文档标题>.md`，解析文档正文内容。

**步骤二：扫描并读取所有图片（必须执行）**

扫描 md 文件中所有图片引用（`![...](...)` 语法），对每一张图片：
1. 用 `read` 工具读取对应的本地图片文件（路径相对于 md 文件所在目录，如 `docs/domain/测试/images/image-01-xxx.png`）
2. 识别图片类型（流程图、架构图、状态机等）并提取关键业务信息
3. 提取的内容标注来源 `[来源：图片 {文件名}]`
4. 无法识别时标注 `[待澄清：图片内容不清晰，请人工说明]`

> **注意**：图片下载到本地后，必须通过 `read` 工具读取本地文件；如果原始图片 URL 需要登录态，直接访问可能返回 401/403。

**步骤三：汇总所有内容进入 Phase 3 规整**

- 获取失败或内容为空：告知用户获取失败，回退到交互式模式（Phase 2.3）

**2.2 从文件获取（--file）**

- 读取指定的本地文件
- 文件不存在时告知用户，回退到交互式模式
- **图片处理**：读取文件内容后，扫描其中的图片引用（Markdown 图片语法 `![...](...)` 或 HTML `<img>` 标签），对每张图片执行以下操作：
  1. 使用 read 工具读取图片文件（支持 PNG、JPG、GIF、SVG 等格式）
  2. 识别图片类型：业务流程图、状态机图、架构图、数据流图、时序图等
  3. 提取图片中的关键信息：节点、流向、状态、角色、系统名称等
  4. 将提取的信息补充到对应维度（流程 → Phase 3 关键业务流程；架构图 → 产品架构；状态机 → 核心业务规则）
  5. 在提取内容旁标注来源 `[来源：图片 {文件名}]`，方便溯源
  6. 若图片无法识别或内容模糊，标注 `[待澄清：图片内容不清晰，请人工补充说明]`

**2.3 交互式引导输入（默认）**

逐维度引导用户补充业务领域知识，每个维度用一次 AskUserQuestion。用户可以对任何维度回复「跳过」。

引导顺序：

**第 1 轮：业务全景**

```
AskUserQuestion:
  question: |
    【业务领域知识注入 1/6 — 业务全景】

    请描述这个系统/产品的业务背景：
    - 这个系统服务于什么业务？解决什么核心问题？
    - 主要用户角色有哪些？（如 C端用户、商家、运营、管理员）
    - 核心业务场景有哪些？（如下单、支付、履约、售后）
    - 各子业务域之间的关系？

    尽量具体，可以用例子说明。回复「跳过」可跳过本维度。
```

**第 2 轮：核心业务规则**

```
AskUserQuestion:
  question: |
    【业务领域知识注入 2/6 — 核心业务规则】

    请列出这个系统中最重要的业务规则：
    - 关键的校验规则（如下单前必须验证库存）
    - 状态流转规则（如订单状态机：待支付→已支付→已发货→已完成）
    - 计算规则（如佣金计算、分账规则、满减规则）
    - 权限规则（如谁能看到什么、谁能操作什么）
    - 约束规则（如优惠券不能与满减叠加、单日限额）

    不需要面面俱到，列出你认为最重要的、代码里不容易看出来的规则。
    回复「跳过」可跳过本维度。
```

**第 3 轮：领域术语**

```
AskUserQuestion:
  question: |
    【业务领域知识注入 3/6 — 领域术语】

    请列出团队/业务中常用的专有术语：
    - 业务概念（如「到店核销」「团购券」「达人探店」）
    - 内部简称或黑话（如 GMV、SKU、POI、LBS）
    - 容易混淆的近义词（如「商家」vs「门店」vs「品牌」）

    格式建议：术语 — 解释
    回复「跳过」可跳过本维度。
```

**第 4 轮：产品架构**

```
AskUserQuestion:
  question: |
    【业务领域知识注入 4/6 — 产品架构】

    请描述产品的整体架构和各端关系：
    - 有哪些端？（如 C端App、B端商家后台、M端运营后台、H5 页面）
    - 各端之间的数据流向和交互关系？
    - 有哪些核心外部依赖？（如支付系统、消息推送、地图服务）
    - 是否有多租户、多地区、多版本等复杂度？

    回复「跳过」可跳过本维度。
```

**第 5 轮：关键业务流程**

```
AskUserQuestion:
  question: |
    【业务领域知识注入 5/6 — 关键业务流程】

    请描述 2-3 个最核心的业务流程（端到端）：
    - 从用户视角描述完整操作路径
    - 涉及哪些角色和系统
    - 关键节点在哪里（如审核、支付、通知）
    - 异常情况怎么处理

    举例：「用户下单流程：浏览商品 → 选择规格 → 加入购物车 → 提交订单 → 
    支付 → 商家接单 → 配送/核销 → 评价」

    回复「跳过」可跳过本维度。
```

**第 6 轮：补充信息**

```
AskUserQuestion:
  question: |
    【业务领域知识注入 6/6 — 补充信息】

    还有哪些你认为 AI 在做需求开发时需要了解的业务背景？
    - 历史遗留问题或技术债
    - 业务上的特殊约定或潜规则
    - 近期业务方向或规划
    - 其他你觉得重要但上面没覆盖到的

    回复「跳过」可跳过本维度。
```
