# Knowledge Extraction Reference

### Phase 3：知识沉淀（除非 --skip-knowledge）

**3.1 新业务规则提炼**

分析本次需求中实现的新业务规则：
- 从 tech_design.md 的接口契约中提取业务约束（如校验规则、状态流转、权限控制）
- 从实际代码中提取隐含的业务规则（如阈值、计算公式、条件分支）
- 与 `.vibe/domain.md` 中已有业务规则对比，只保留新增的
- 新业务规则写入 `.vibe/domain.md` 对应章节（经人工确认后）
- 新领域术语同步追加到 domain.md 的术语表

每个新规则格式：
- **规则描述**：一句话
- **来源**：哪个文件/方法中实现的
- **类型**：数据校验 / 状态流转 / 权限控制 / 计算规则 / 其他

**3.2 新代码模式提取**

检查本次开发中是否出现了仓库中尚未记录的新模式：
- 新的链路模式（如首次实现的异步消费链路）
- 新的组件模式（如新的缓存策略、新的外部调用封装）
- 新的工具方法（如新加入的通用转换函数）

提取标准：
- 至少在本次代码中出现 2 处以上，或明确具有通用复用价值
- 符合 Day0 模板的格式要求（适用场景、骨架步骤、固定/可替换步骤）
- 不足 2 个样本的模式标注 `[待观察]`，不生成模板

**3.3 AGENTS.md 更新候选**

对比现有 AGENTS.md，识别需要更新的工程规范：
- 新的构建/测试命令（如有变化）
- 新的模块说明（如新增模块）
- 新的命名规则（如发现了新的命名约定）
- 新的禁止模式（如发现了需要避免的写法）
- 新的硬规则（由多个样本支撑的新规则）

**3.4 domain.md 更新候选**

对比现有 `.vibe/domain.md`，识别需要更新的业务知识：
- 新的业务规则（校验、状态流转、计算、权限、约束）
- 新的领域术语（本次需求引入的新概念）
- 业务流程更新（本次改动影响的流程变化）
- 产品架构更新（如新增了外部依赖或端关系变化）
